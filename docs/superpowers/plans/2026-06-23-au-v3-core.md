# AU v3 Async Instantiation (Plan A) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** host **AU v3** plugins' audio (effects **and** instruments) by adding **async instantiation**
(`AudioComponentInstantiate`) to `AudioUnitFormat.Load`, routing v3 / async-required components through the
**existing, unchanged `AudioUnitPlugin`**. The Task 0 spike proved the shipped v2 C-API path drives v3 AUs over
Apple's bridge — in- and out-of-process — so **no new plugin class and no `AUAudioUnit` Obj-C object are needed
for audio/params/state/MIDI**.

**Architecture (post-spike, simplified):** `AudioComponentFindNext` already finds v3 components and the
`kAudioComponentFlag_IsV3AudioUnit` / `RequiresAsync` / `CanLoadInProcess` flags. The **only** change is in
`Load`: for a component that requires async (all real v3 AUs do), call `AudioComponentInstantiate(comp, options,
completionBlock)` — `options = LoadOutOfProcess` when `CanLoadInProcess` is clear, else `LoadInProcess` — await
the one completion block, and hand the resulting `AudioComponentInstance` to the **same** `AudioUnitPlugin`. That
plugin's existing render shim, parameter C-API, `ClassInfo` state, and `MusicDeviceMIDIEvent` all work over the
bridge (spike-verified). The lone new interop is a single global completion block (`AuBlocks.MakeCompletionBlock`).

**Spec:** `docs/superpowers/specs/2026-06-23-au-v3-hosting-design.md`. **The v3 editor is NOT here** — `CocoaUI`
is absent for v3 AUs, so the editor needs the `AUAudioUnit` view-controller path → **Plan B**
(`…au-v3-editor.md`).

**Tech Stack:** C# (net10.0), AudioToolbox C API + one hand-built Obj-C completion block, xUnit (discovery) +
the main-thread `MacEditorHarness` (load/render/state). **arm64, macOS-only.**

> **⚠ Bring-up finding (2026-06-23):** `AudioComponentInstantiate`'s completion is delivered on the **main
> dispatch queue**, so the run-loop pump in `Load` only sees it on the thread draining that queue. The worker
> calls `Load` on its **main thread** (thread 0, `Worker/Program.cs`), so production is correct — but **xUnit runs
> on pool threads and hangs**. So, exactly as with the Cocoa editor: **discovery is verified in xUnit; async
> load + OOP render + instrument note + ClassInfo state are verified on thread 0 by `MacEditorHarness`** (arms
> `RenderAuV3` for `DimChorus` OOP-effect and `AudMod` in-proc-instrument). ALL TASKS DONE ✅.

## Global Constraints

- **`dotnet`**, net10.0, **0/0**, dependency-free, **additive** — the v2 sync-load path and `AudioUnitPlugin` are
  unchanged; v3 only adds an async-load branch in `Load`.
- `Process` stays realtime-clean (it already is — `AudioUnitPlugin` is reused verbatim).
- macOS-only tests self-skip off-OS and skip when no v3 AU is installed (so CI/other machines stay green); on
  this Mac they run for real against `DimChorus` (effect) and `AudMod` (instrument).

## Task 0 — Async/OOP + bridge spike  — **gate** ✅ SPIKE PASS 2026-06-23

- [x] Constructed a global Obj-C completion block and called `AudioComponentInstantiate` for **`DimChorus`**
      (`LoadOutOfProcess`) and **`AudMod`** (`LoadInProcess`); both delivered an `AudioComponentInstance`.
- [x] Drove each with the **v2 C API**: stereo non-interleaved-float ASBD, input render callback (effects),
      `AudioUnitInitialize`, `AudioUnitRender`, `MusicDeviceMIDIEvent` (instrument). All worked over the bridge.
- [x] Measured OOP render latency.

> **Outcome (2026-06-23, scratchpad `au-v3-spike`):** SPIKE PASS. `AudioComponentInstantiate` (async, one
> completion block) + the **existing v2 C API** drives v3 AUs both in- and out-of-process:
> - **DimChorus** (OOP effect): instantiated 357 ms; `AudioUnitRender` works (chorus alters the signal, out RMS
>   0.3337, diff-from-dry 0.1168); **3 params via the v2 ParameterList**; **OOP render latency avg 18 µs / max
>   23 µs** against a 10.67 ms block — i.e. **0.2%; the XPC-latency concern is moot** (shared-memory bridge).
> - **AudMod** (in-proc instrument): instantiated 156 ms; a note via **`MusicDeviceMIDIEvent`** renders audio
>   (RMS 0.0559); 124 params.
> - **`kAudioUnitProperty_CocoaUI` is *absent* for both** → the editor is the only genuinely-v3 piece (Plan B).
> - Completion block layout confirmed: global block, `isa=_NSConcreteGlobalBlock` (via `dlsym` on libSystem),
>   flags `BLOCK_IS_GLOBAL (0x10000000)`, `invoke @16`, descriptor `{reserved=0,size=32} @24`.
>
> **Design collapsed:** no `AudioUnitV3Plugin`, no `renderBlock`/`pullInputBlock`/`AUParameterTree`/`fullState`
> Obj-C, no `scheduleMIDIEventBlock`. Plan A is now just the async-load branch + a one-block helper. Tasks below
> rewritten accordingly.

## Task 1 — Completion-block helper (`AuBlocks.cs`)  ✅

- [x] `AuBlocks.MakeGlobalBlock(invokeFnPtr)` builds a non-capturing global block
      `{ isa=_NSConcreteGlobalBlock; flags=BLOCK_IS_GLOBAL; reserved; invoke; descriptor={0,32} }` (spike-proven
      layout; `_NSConcreteGlobalBlock` via `NativeLibrary` on `/usr/lib/libSystem.B.dylib`). A small
      `AudioComponentInstantiate` P/Invoke + the `AudioComponentInstantiationOptions` constants go in
      `AudioUnitAbi`. Builds 0/0.

## Task 2 — Async-load branch in `AudioUnitFormat.Load`  ✅

- [x] In `Load`, read the component's flags (`AudioComponentGetDescription`). `RequiresAsyncLoad` → async when
      `RequiresAsync`, or a v3 AU that can't load in-process; otherwise the legacy v2 sync path is untouched.
      Async calls `AudioComponentInstantiate(comp, CanLoadInProcess ? LoadInProcess : LoadOutOfProcess, block)`,
      pumps `CoreFoundation.PumpRunLoop` until the `[UnmanagedCallersOnly]` completion sets the handshake, and
      hands the `AudioComponentInstance` to the **unchanged** `AudioUnitPlugin`. (Discovery unchanged.) A static
      gate serializes the rare loads against the one process-global completion block.
- [x] **Discovery test** (xUnit) `AudioUnitV3Tests.Discovers_the_dimchorus_v3_effect`; load itself is proven in
      the harness (see finding above — async completion is main-queue, so xUnit can't drive it).

## Task 3 — Verify v3 effect audio (the existing render shim, over the bridge)  ✅
