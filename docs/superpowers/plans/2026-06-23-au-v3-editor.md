# AU v3 Custom-UI via the AUAudioUnit Front-End (Plan B) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** host AU v3 plugins through the **`AUAudioUnit` Objective-C front-end** so a single instance serves both
**audio** and the plugin's **own custom view**. This is a deliberate re-architecture chosen over the generic-view
fallback (see Task 0): v3 AUs vend no `kAudioUnitProperty_CocoaUI`, and the custom view comes only from
`AUAudioUnit`'s `requestViewControllerWithCompletionHandler:` — which, for out-of-process AUs, **cannot share**
the v2 `AudioComponentInstance` Plan A used (its `audioUnit` bridge is null OOP). So to get the real UI bound to
the running instance, v3 audio must also be driven from the `AUAudioUnit` (`renderBlock` / `parameterTree` /
`fullState` / `scheduleMIDIEventBlock`).

**Supersedes Plan A's v3 audio path.** Plan A routed v3 components to the v2 `AudioUnitPlugin` over
`AudioComponentInstantiate`. That path is **replaced for v3** by a new `AudioUnitV3Plugin` here. Reused from Plan
A: `AuBlocks` (the global completion block), `CoreFoundation.PumpRunLoop`, the component-flag reads, discovery,
and the AU-v2 sync path (legacy v2 AUs are **untouched**).

## Task 0 — Front-end spike (editor + render)  — **gate** ✅ SPIKE PASS 2026-06-23

- [x] **Editor probe** (`scratchpad/au-v3-editor-spike`): `AUGenericView initWithAudioUnit:(v3 instance)` works
      (so the generic editor is the cheap fallback) **but** the *custom* view needs `AUAudioUnit`. The
      `AUAudioUnit` front-end gives `providesUserInterface=true` and `requestViewControllerWithCompletionHandler:`
      → a real `NSView` (DimChorus 500×210). **Critically, `[AUAudioUnit audioUnit]` is null for an OOP AU** → the
      custom view can't ride Plan A's v2 instance → re-architecture required.
- [x] **Render probe** (`scratchpad/au-v3-render-spike`): drove DimChorus audio from the **`AUAudioUnit`
      `renderBlock`**: set input+output bus formats to a standard `AVAudioFormat` (deinterleaved float32 = our
      planar bus), `maximumFramesToRender`, `allocateRenderResourcesAndReturnError:`, then called
      `renderBlock(flags, ts, frames, bus, outABL, pullInputBlock)` with a constructed pull-input block. Output
      non-silent and chorus-altered (RMS 0.3337, diff 0.1168); **per-block latency avg 21µs / max 29µs of a
      10.67ms block** — negligible.

> **Outcome:** the whole front-end is proven — async instantiate (reuses `AuBlocks`), bus-format negotiation,
> `renderBlock` realtime audio with a pull-input block, and the custom view controller. Remaining pieces are
> mechanical Obj-C (`parameterTree`, `fullState`, `scheduleMIDIEventBlock`). Build `AudioUnitV3Plugin`.

## Architecture

A new `AudioUnitV3Plugin : IHostedPlugin, IPluginGui` (peer to `AudioUnitPlugin`) plus an `AuV3` interop helper
(peer to `AuAppKit`, holding the `objc_msgSend` slices + the constructed pull-input/MIDI blocks). `AudioUnitFormat`
routes async/v3 components here instead of to `AudioUnitPlugin`. The Cocoa `EditorHost`, `EditorSession`, and the
neutral-background fix are **unchanged** — the v3 view rides them exactly as the v2 view does, proving the seam.

**Tech Stack:** C# (net10.0), AudioToolbox + AVFoundation + AppKit `objc_msgSend` + hand-built Obj-C blocks
(`AuBlocks`), xUnit (discovery) + the main-thread `MacEditorHarness` (load/render/state/editor). **arm64,
macOS-only.** Async instantiation + the editor are **main-thread** (completion lands on the main dispatch queue),
so verification is the harness on thread 0, as with Plan A.

## Global Constraints

- **`dotnet`**, net10.0, **0/0**, dependency-free, **additive** where possible. The v2 `AudioUnitPlugin` and the
  AU-v2 sync path stay byte-for-byte; only v3 routing changes. **No change to `EditorSession`/the Cocoa backend.**
- `Process` stays realtime-clean: the `renderBlock` and pull-input block are cached; no per-block allocation.
